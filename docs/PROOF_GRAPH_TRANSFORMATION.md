# ProofArena 超级改造纲领

> 目标不是做一个更漂亮的搜题网站，而是开创一个新的数学学习品类：把数学解法变成可比较、可挑战、可验证、可积累的推理网络。

本文定义 ProofArena 下一阶段的产品方向、技术架构、内容标准、用户体验标准和阶段验收要求。所有后续功能都应服从这份纲领：如果一个改动不能强化“数学推理网络”，就不应成为核心投入。

## 1. 根本定位

### 1.1 ProofArena 是什么

ProofArena 是一个数学推理竞技场。

它不是：

- 题目搜索工具
- 答案搬运站
- AI 拍题解答器
- 普通题库加评论区
- 普通竞赛平台

它是：

- 把同一道题的不同解法放在同一个场域中正面比较的系统
- 把解法背后的观察、分支、转化、边界和失败路线结构化保存的系统
- 让用户通过提交、挑战、审核、评分、讨论形成个人数学身份的系统
- 让平台内容越积累越难复制、越使用越有价值的数学推理网络

### 1.2 一句话愿景

ProofArena 让数学解法从“答案文本”升级为“可比较、可挑战、可验证、可传承的推理资产”。

### 1.3 与搜题软件的根本差异

普通搜题软件的链路：

```text
遇到题目 -> 搜索/拍照 -> 得到答案 -> 离开
```

ProofArena 的链路：

```text
遇到题目 -> 进入解法竞技场 -> 比较路线 -> 理解选择原因 -> 提交/挑战/收藏/讨论 -> 形成个人解法身份 -> 持续回来
```

普通平台的核心资产是题量和答案覆盖率。

ProofArena 的核心资产是：

- 解法之间的关系
- 推理路径的结构化表达
- 方法边界和误用记录
- 用户挑战和贡献网络
- 审核后沉淀的高质量数学推理图谱

这些资产无法通过低成本添加一个“多解法”按钮复制。

## 2. 产品护城河

### 2.1 第一层：结构化内容资产

ProofArena 不应只保存 Markdown 解法。每条高质量解法都应逐步拆成结构化对象：

- 入口观察：为什么会想到这条路
- 题面信号：哪个条件触发了这个方法
- 思路分支：还有哪些可选方向
- 关键转化：真正改变问题形态的一步
- 解法路径：完整可复算步骤
- 验证步骤：可代入、可数值检查、可 CAS 验证的位置
- 方法边界：这个方法什么时候适用，什么时候不该用
- 失败路线：看似合理但会绕远或失败的方法
- 挑战关系：它挑战了哪条解法，强在哪里，弱在哪里
- 迁移关系：这条观察还能迁移到哪些题

长期看，ProofArena 不是题库，而是数学推理图谱。

### 2.2 第二层：交互式解法引擎

用户不只是阅读解法，而是操作解法。

核心交互包括：

- 按“考场 / 复盘 / 讲解 / 欣赏 / 挑战”切换视角
- 横向比较多条解法
- 点击关键转化查看它来自哪些观察
- 点击方法边界查看它在哪些题里失败或成立
- 点击挑战关系查看双方差异
- 点击用户查看他的解法风格和代表作品
- 从一个观察跳到相似观察的其他题

这会让 ProofArena 成为推理操作系统，而不是静态内容页。

### 2.3 第三层：审核与验证生产线

ProofArena 的内容质量不靠随缘投稿，而靠系统化生产流程。

理想链路：

```text
用户提交自然语言解法
  -> 系统辅助拆解结构字段
  -> CAS/规则检查关键步骤
  -> 相似解法检测
  -> 审核员补评分、边界、标签、挑战关系
  -> 发布为正式解法
  -> 自动进入 Proof Graph
  -> 后续被评分、挑战、引用、迁移
```

AI 和 CAS 不能作为表面卖点，而要嵌入投稿、审核、发布和阅读全流程。

## 3. Proof Graph 核心架构

### 3.1 领域对象

Proof Graph v1 至少包含：

```text
Problem
Solution
Observation
StrategyBranch
Transformation
VerificationStep
MethodBoundary
ChallengeEdge
UserContribution
UserProfile
```

后续可扩展：

```text
ConceptNode
MisusePattern
TransferEdge
SimilarityEdge
ContestEntry
ReviewDecision
LearningPath
```

### 3.2 对象定义

#### Problem

题目不是终点，而是推理网络的入口。

必须支持：

- 基础来源信息
- 题干和答案
- 标签和难度
- 学习指南
- 解法集合
- 思路图谱
- 概念边界
- 相关题和迁移路径

#### Solution

解法不是文章，而是一条可比较的路线。

必须支持：

- 类型：标准解、启发解、稳健解、教学解等
- 作者和来源
- 入口观察
- 关键转化
- 完整步骤
- 五维评分
- 适用场景
- 代价与局限
- 验证状态
- 挑战关系

#### Observation

Observation 记录“读题时先看见什么”。

好 Observation：

- 指向具体条件
- 能解释为什么触发某种方法
- 能迁移到其他题

坏 Observation：

- “认真审题”
- “观察结构”
- “想到函数思想”

#### StrategyBranch

StrategyBranch 记录一个题目中可能分化出的路线。

它必须回答：

- 这个分支从哪个观察开始
- 它可能通向哪些方法
- 它为什么值得尝试
- 它为什么可能失败
- 它关联哪些正式解法

#### Transformation

Transformation 是解法中真正改变问题形态的一步。

它必须回答：

- 从什么表达变成什么表达
- 为什么这一步合法
- 这一步降低了什么复杂度
- 它适合哪些相似问题

#### VerificationStep

VerificationStep 是可复核点。

它可以是：

- 代入检查
- 边界检查
- 取等检查
- 数值抽样
- CAS 检查
- 图形验证
- 独立人工复算

#### MethodBoundary

MethodBoundary 记录方法边界。

必须包含：

- 方法名称
- 在本题为什么不优先用
- 什么条件下它会变好
- 常见误用
- 相关概念

这是 ProofArena 区别于普通题解的关键模块。

#### ChallengeEdge

ChallengeEdge 记录解法之间的竞技关系。

字段应包含：

- challengerSolutionId
- targetSolutionId
- claim：一句话说明挑战点
- advantages：更短、更稳、更少计算、更好讲、更有洞察等
- risk：在哪些方面不如原解
- reviewerNote：审核员对挑战是否成立的判断
- publicStats：用户投票和使用偏好

#### UserContribution

用户贡献不是简单投稿记录，而是个人数学身份的来源。

它应沉淀：

- 已发布解法
- 挑战记录
- 被挑战记录
- 被收藏和引用记录
- 评分表现
- 擅长知识点
- 解法风格画像

### 3.3 存储策略

短期继续使用 Next.js + Supabase + Postgres，不重开项目，不换技术栈。

建议策略：

- 稳定核心实体使用关系表
- 富结构内容使用 JSONB
- 关系边使用独立表
- 静态 fallback 继续保留，但字段契约必须同步
- 后续需要相似度时再引入 `pgvector`
- 暂不引入图数据库，避免过早复杂化

优先级：

1. 先把领域模型建正确
2. 再把 UI 体验跑通
3. 再做智能化和推荐

## 4. 核心产品模块标准

### 4.1 题目页标准

题目页必须成为 ProofArena 的主战场。

用户进入题目页后，应清楚看到：

- 题目本身
- 为什么这题值得看
- 有哪些思路入口
- 每条解法适合什么场景
- 解法之间如何比较
- 哪些方法不该优先用
- 如何提交自己的解法挑战

题目页不应变成：

- 长答案列表
- 无重点百科页
- 只展示静态 Markdown 的阅读页

### 4.2 解法比较矩阵

每道有多解法的题，都应逐步支持解法比较矩阵。

矩阵至少包含：

| 维度 | 说明 |
| --- | --- |
| 正确性 | 是否可靠、边界是否完整 |
| 考场性 | 时间压力下是否可执行 |
| 结构美感 | 是否抓住本质结构 |
| 计算量 | 展开、分类、重复运算是否可控 |
| 讲解友好 | 是否适合复盘和教学 |
| 风险 | 容易卡在哪里 |
| 最适合用户 | 考生、教师、竞赛爱好者、复盘者 |

用户应能切换视角：

- 我在考场
- 我在复盘
- 我要讲给别人
- 我要学习迁移
- 我要找更优雅的路线

### 4.3 思路图谱

思路图谱不是装饰图，而是推理导航。

它必须展示：

- 入口观察
- 分支路线
- 关键转化
- 对应解法
- 失败或绕远路线
- 方法边界

低质量思路图谱的表现：

- 只是把解法标题画成节点
- 没有解释分支为什么出现
- 默认折叠后只剩标题
- 移动端无法阅读

### 4.4 为什么不用这个方法

每道重点题应有“为什么不用这个方法”模块。

模块必须回答：

- 这个方法为什么看起来诱人
- 在本题为什么不优先
- 它会在哪一步变复杂
- 什么条件出现时它会变成好方法
- 有没有相关反例题或迁移题

这类内容是 ProofArena 的差异化核心，不应被视为附属说明。

### 4.5 数学推理回放

推理回放展示一条解法如何被发现，而不是只展示完成后的答案。

标准结构：

```text
看到什么条件
  -> 可能想到哪些路线
  -> 排除哪些路线
  -> 为什么某条路线变好
  -> 关键转折是什么
  -> 如何验证它没有走偏
```

推理回放的目标是让用户产生“我下次也可能想到”的感觉。

### 4.6 解法挑战

解法挑战必须比普通评论更严格。

用户提交挑战时必须选择：

- 挑战对象
- 自己强在哪里
- 优势标签
- 风险自评

审核通过后必须进入：

- 题目页解法卡片
- 解法比较矩阵
- 个人公开主页
- 挑战关系图

挑战不是吵架，也不是点赞。它是结构化比较。

### 4.7 个人解法档案

个人主页不只是用户资料，而是数学身份。

必须逐步支持：

- 已发布解法
- 代表解法
- 挑战记录
- 被挑战记录
- 作品均分
- 代表优势
- 擅长知识点
- 解法风格画像
- 被收藏和引用

长期目标：

用户愿意把 ProofArena 公开主页当成自己的数学作品集。

### 4.8 投稿与审核

投稿表单不应只收一段文本。

它应引导用户表达：

- 为什么想到
- 关键转化
- 完整步骤
- 可验证位置
- 适用场景
- 与已有解法的关系
- 风险与局限

审核后台必须支持：

- 结构化编辑
- 公式转码
- 五维评分
- 真实卡片预览
- 挑战信息修正
- 审核评语
- 发布到题库

审核不是门禁，而是把用户投稿转化成长期内容资产的生产线。

## 5. AI 与 CAS 使用原则

### 5.1 AI 不做表面入口

不要把 ProofArena 变成“又一个 AI 搜题框”。

AI 应该嵌入系统内部：

- 投稿时辅助拆结构
- 审核时指出缺失字段和潜在漏洞
- 比较时总结解法差异
- 阅读时按用户目标调整解释粒度
- 复盘时分析用户为什么没想到关键步骤
- 推荐时基于观察和转化匹配下一题

### 5.2 CAS 是验证层，不是替代证明

CAS 负责：

- 代数化简检查
- 方程求解核对
- 导数和极值检查
- 数值抽样
- 边界候选验证

CAS 不应宣称：

- 自动完成全部证明
- 替代人工审核
- 保证所有推理严密

页面必须区分：

- verified
- partial
- manual

### 5.3 智能审核优先级

智能化先用于审核和内容生产，不先用于用户端炫技。

优先做：

1. 草稿转结构字段
2. 相似解法检测
3. 步骤漏洞提示
4. 评分建议
5. 方法边界建议

## 6. 多端策略

### 6.1 现阶段不重开项目

继续基于现有 Next.js + Supabase 项目改造。

不建议现在：

- 重写后端
- 换图数据库
- 做原生 App
- 做 Electron 桌面端
- 全力做小程序
- 把主体验搬到 IM 机器人

原因：

- 当前项目已有题目页、解法、投稿、审核、个人主页、比赛和 CAS 雏形
- Proof Graph 可以在现有架构上渐进式落地
- 当前最缺的是不可替代的核心体验，不是更多入口

### 6.2 各端长期定位

网页：

- 主体验
- 深度阅读
- 解法比较
- 思路图谱
- 审核后台
- 公开个人主页

小程序：

- 分享入口
- 每日题
- 轻量浏览
- 通知和提醒
- 邀请赛传播

IM 机器人：

- 投稿提醒
- 审核通知
- 比赛开题通知
- 快速收集草稿
- 群内引流回网页

独立软件：

- 暂缓
- 只有当需要本地 CAS、手写草稿、离线题库或课堂投屏时再考虑

## 7. 内容标准

### 7.1 高质量解法标准

一条正式解法必须尽量满足：

- 路线有实质差异
- 能说明为什么想到
- 关键转化明确
- 步骤可复算
- 适用场景清楚
- 代价与局限诚实
- 有评分理由
- 有验证状态
- 可与其他解法比较

### 7.2 不接受的内容

不应作为正式优质解法：

- 只写答案
- 只写“显然”
- 大段复制第三方解析
- 与已有解法只有符号差异
- 没有关键步骤
- 没有说明适用场景
- 无法复核来源

### 7.3 重点内容优先级

ProofArena 应优先积累：

1. 一题多解明显的题
2. 容易误用方法的题
3. 有漂亮关键转化的题
4. 能体现考场解与优雅解差异的题
5. 能形成用户挑战的题
6. 能迁移到一组题的观察

题量不是第一指标。少量高密度题目比大量普通答案更重要。

## 8. 用户体验标准

### 8.1 第一屏标准

用户进入核心页面后，必须很快明白：

- 这里不是搜答案
- 这里在比较解法
- 我可以参与挑战
- 我的贡献会留下来

### 8.2 阅读节奏

ProofArena 应采用渐进式披露：

- 先看题
- 再看思路入口
- 再看比较
- 再展开完整解法
- 再看边界、验证和讨论

不要默认把所有长内容堆在一起。

### 8.3 移动端标准

移动端必须保证：

- 题目可读
- 解法卡片按钮可点
- 横向切换和纵向阅读不冲突
- 比较矩阵可压缩或分组
- 长公式不撑破布局
- 公开主页可分享

### 8.4 视觉气质

ProofArena 应该克制、清晰、专业。

避免：

- 大面积霓虹科技风
- 过度毛玻璃
- 装饰性渐变堆叠
- 只为炫技的动效
- 营销式空洞文案

适合：

- 深度阅读
- 高质量内容
- 清楚的信息层级
- 稳定的数学排版
- 类似作品集和研究笔记的可信感

## 9. 成功指标

### 9.1 内容指标

早期核心指标：

- 每道重点题至少 3 条实质不同解法
- 每条正式解法至少有入口观察和关键转化
- 每道重点题至少 1 个方法边界
- 每道重点题至少 1 条可比较挑战关系
- 每周至少沉淀 1 条可长期展示的用户解法

### 9.2 用户指标

比 DAU 更重要的早期指标：

- 有多少用户提交第二条解法
- 有多少用户挑战已有解法
- 有多少用户分享自己的公开主页
- 有多少题被用户收藏为“以后复盘”
- 有多少讨论能被整理进正式内容

### 9.3 质量指标

必须追踪：

- 审核通过率
- 退回修改率
- 平均审核耗时
- 发布后被收藏数
- 被挑战次数
- 被引用到其他题的次数

### 9.4 护城河指标

证明平台壁垒形成的信号：

- 用户说“这里能看到为什么不用某方法”
- 用户说“这里能比较哪条解法适合考场”
- 用户愿意把公开主页发给别人
- 教师愿意引用 ProofArena 的路线比较
- 新投稿能复用旧观察、转化和边界节点
- 一条新解法能自动进入已有推理网络

## 10. 阶段路线

### Phase 0：现有能力巩固

已经具备或正在具备：

- 多解法题目页
- 五维评分
- 解法挑战
- 投稿审核
- 公开个人主页
- 比赛雏形
- CAS 验证雏形

必须保持：

- `npm run lint` 通过
- `npm run build:webpack` 通过
- Supabase migration 顺序清晰
- 静态 fallback 不破坏

### Phase 1：Proof Graph v1

目标：把内容结构从“解法数组”升级为“推理图谱”。

范围：

- Observation
- StrategyBranch
- Transformation
- VerificationStep
- MethodBoundary
- ChallengeEdge

验收：

- 至少 3 道重点题拥有完整 Proof Graph v1 数据
- 题目页能展示思路图谱
- 解法比较矩阵能读取结构字段
- 投稿审核能编辑至少部分 Proof Graph 字段

### Phase 2：核心题目页重构

目标：让题目页成为独特体验，而不是答案列表。

范围：

- 题目阅读区
- 思路入口区
- 解法比较矩阵
- 解法擂台
- 为什么不用这个方法
- 推理回放

验收：

- 新用户 30 秒内能理解 ProofArena 和普通题库不同
- 移动端可完整走完“看题 -> 比较 -> 展开解法 -> 挑战”路径
- 至少 1 道题达到可公开展示级别

### Phase 3：用户身份网络

目标：让用户有留下来的理由。

范围：

- 公开主页加强
- 代表解法
- 风格画像
- 挑战/被挑战关系
- 收藏和引用
- 用户贡献时间线

验收：

- 用户能分享个人主页作为数学作品集
- 题解作者名可点击进入公开主页
- 个人页能展示“我擅长什么、我贡献了什么、我挑战了什么”

### Phase 4：智能审核和辅助生产

目标：让高质量内容生产效率上升。

范围：

- 投稿草稿结构化
- 相似解法检测
- CAS 检查建议
- 方法边界建议
- 审核评分辅助

验收：

- 审核员能更快把自然语言投稿变成正式题解
- AI 输出必须可编辑、可拒绝、可追踪
- 不允许 AI 直接绕过人工审核发布

### Phase 5：传播与多端入口

目标：在核心体验成立后扩大入口。

范围：

- 小程序轻量入口
- IM 机器人通知和投稿草稿入口
- 比赛邀请传播
- 公开主页分享卡

验收：

- 多端只做入口，不复制完整主体验
- 所有入口最终回流到 ProofArena 网页主体验
- 不因多端破坏核心产品节奏

## 11. 技术标准

### 11.1 不重开项目

本轮超级改造基于现有项目渐进式升级。

允许：

- 重构模块
- 新增领域层
- 新增数据库表
- 调整页面结构
- 迁移静态数据

不建议：

- 从零重写
- 为了技术新鲜感换栈
- 过早引入复杂微服务
- 过早引入图数据库

### 11.2 数据库标准

每个 migration 必须：

- 编号唯一
- 可重复执行的地方使用 `IF NOT EXISTS`
- 不破坏已有数据
- RLS 策略明确
- 公共数据和私有数据边界清楚

公开可读：

- 正式题目
- 正式解法
- 公开用户资料
- 公开评论和评分

不公开：

- 未审核投稿
- 审核过程中的内部判断
- 用户邮箱
- 私有草稿

### 11.3 前端标准

核心页面必须：

- 类型检查通过
- 生产构建通过
- 移动端可用
- 数学公式不泄露原始 LaTeX
- 关键按钮有明确状态
- 空状态不尴尬
- 不用大段功能说明替代真实功能

### 11.4 验证标准

每次核心改动至少运行：

```bash
npm run lint
npm run build:webpack
```

涉及移动端交互、图形、长公式、复杂页面时，必须做窄屏浏览器检查。

涉及数据库字段时，必须检查：

- migration
- 类型定义
- mapper
- 投稿写入
- 审核读取和保存
- 发布逻辑
- 公开页面读取

## 12. 决策原则

### 12.1 优先做能积累护城河的功能

高优先级：

- 结构化解法
- 解法关系
- 方法边界
- 用户贡献沉淀
- 审核生产线
- 推理图谱

低优先级：

- 普通刷题打卡
- 普通搜索页优化
- 泛论坛
- 无结构 AI 问答
- 单纯换 UI 风格

### 12.2 不追求题量虚胖

ProofArena 早期宁可只有少量题，也要每道题足够深。

一题应该像一个小型研究场：

- 有多个路线
- 有比较
- 有边界
- 有挑战
- 有用户贡献
- 有迁移价值

### 12.3 所有功能都要回到核心路径

核心路径：

```text
看题 -> 比较不同解法 -> 理解路线选择 -> 提交自己的解法 / 挑战已有解法 -> 沉淀到个人档案和推理图谱
```

不服务这条路径的功能，默认不是优先级。

## 13. 掀桌子级设计原则

ProofArena 不应把自己放在“搜题软件、题库、AI 解题工具、学习社区”的同一条赛道上比较。那样即使做得更好，也只是同质化竞争。

ProofArena 要做的是品类重定义：让用户意识到“找答案”本身是一种落后的数学学习方式。真正值得追求的是理解路线、比较路线、创造路线、公开接受挑战，并把自己的推理能力沉淀成可见资产。

### 13.1 反搜题，不是更强搜题

普通产品默认用户想要答案。ProofArena 应默认用户想要变强。

这意味着：

- 不以“最快给出最终答案”为最高目标
- 不把完整答案作为唯一核心产物
- 不让用户停留在复制步骤
- 不奖励无解释的答案堆叠
- 不把“题量覆盖”当作品牌核心

ProofArena 应鼓励：

- 先判断路线
- 再比较路线
- 再展开证明
- 再挑战和改写
- 最后沉淀成个人作品

判断标准：

如果一个新功能只是让用户更快拿到答案，它不是 ProofArena 的核心功能。

### 13.2 把数学解法做成 Git，而不是文章

ProofArena 可以借鉴 Git 的思想，但应用在数学推理上。

每条解法都不只是内容，而是一个可 fork、可 diff、可 challenge、可 merge 的推理版本。

未来应逐步支持：

- Fork：基于某条解法改写成新路线
- Diff：显示两条解法真正不同的步骤
- Pull Request：用户提交改进，审核员合并到正式解法
- Blame/Provenance：每个关键观察和转化有来源
- Branch：同一入口分化出多个方法分支
- Merge：多个用户贡献整理成一条高质量正式解法
- Release：一场比赛或专题形成稳定解法集

这会形成普通题库完全没有的产品心智：

> 数学解法不是一次性答案，而是可协作演化的推理版本。

### 13.3 把题目做成世界，不是卡片

普通平台的一道题是一张卡片。ProofArena 的一道题应该像一个可探索的小世界。

一题内部应包含：

- 地图：思路图谱
- 路线：解法路径
- 岔路：策略分支
- 禁区：不该优先用的方法
- 关卡：关键转化
- 证据：验证步骤
- 竞技场：解法挑战
- 居民：贡献过这题的用户
- 历史：这道题的解法演化过程

用户应该愿意在一道好题里停留很久，而不是看完答案马上走。

### 13.4 从“答案正确”升级到“路线选择正确”

搜题软件解决的是：

> 这道题答案是什么？

ProofArena 要解决的是：

> 面对这道题，我为什么应该选择这条路线？

这是更高阶的问题，也更难被低成本模仿。

因此所有题解都应逐步回答：

- 这条路线什么时候最值得用
- 这条路线在什么时间压力下可行
- 这条路线对什么水平的学生友好
- 它比另一条路线强在哪里
- 它在哪些地方可能输给另一条路线
- 它暴露了题目背后的什么结构

一条只给正确步骤、不解释路线选择的解法，在 ProofArena 中只能算半成品。

### 13.5 建立数学声誉协议

用户留下来，不是因为平台有积分，而是因为这里能沉淀真实的数学声誉。

ProofArena 应把个人主页升级成一种“数学履历协议”：

- 我创造过哪些解法
- 我挑战过哪些经典路线
- 哪些解法被别人采用
- 我在哪些题型上有优势
- 我的风格更像考场型、洞察型、教学型还是结构型
- 我的贡献被哪些题目、专题、比赛引用
- 我在哪些推理节点上有原创贡献

这比普通等级、徽章、连续打卡更高级。

长期目标：

> 一个用户愿意把 ProofArena 主页发给老师、同学、社群甚至申请材料，因为它展示的是他的数学思维作品，而不是刷题数量。

### 13.6 从内容平台升级为推理协议

如果 ProofArena 只是一组页面，别人可以模仿页面。

如果 ProofArena 是一套数学推理表达协议，别人就必须重建整个数据、审核、交互和社区系统。

ProofArena 应逐步定义自己的推理协议：

```text
Observation -> Branch -> Transformation -> ProofStep -> Verification -> Boundary -> Challenge -> Contribution
```

每个正式解法都应尽量符合这套协议。

这套协议的价值：

- 可以驱动页面展示
- 可以驱动审核流程
- 可以驱动用户画像
- 可以驱动相似题推荐
- 可以驱动 AI 辅助
- 可以驱动 CAS 验证
- 可以驱动比赛复盘
- 可以驱动内容导出和分享

当协议成立后，ProofArena 就不再只是一个网站，而是数学推理内容的组织方式。

### 13.7 让 AI 成为幕后引擎，而不是聊天入口

AI 搜题很容易同质化。ProofArena 要避免把 AI 放在最表层。

更前瞻的方向是：

- AI 负责把草稿变成结构化推理协议
- AI 负责比较两条解法的真实差异
- AI 负责找出缺失的边界和验证点
- AI 负责在审核后台提出风险提示
- AI 负责根据用户风格推荐下一条挑战
- AI 负责把比赛投稿整理成赛后推理图谱

用户感知到的不是“一个会聊天的 AI”，而是：

> 这个系统真的理解解法之间的关系。

这比普通 AI 搜题更难做，也更难被抄。

### 13.8 让每次使用都增加系统智能

普通题库使用一次，平台没有变强。

ProofArena 每次使用都应该尽量让系统变强：

- 用户收藏一条解法，增加“适用场景”信号
- 用户挑战一条解法，增加 ChallengeEdge
- 用户评分一条解法，增加比较数据
- 用户评论被采纳，转化成 MethodBoundary 或 Observation
- 审核员发布解法，增加结构化节点
- 比赛结束，增加一组高质量路线关系

最终形成数据飞轮：

```text
高质量题目 -> 多路线比较 -> 用户挑战 -> 审核沉淀 -> 推理图谱增强 -> 阅读体验增强 -> 更多高质量用户参与
```

这个飞轮才是长期领先的根。

### 13.9 创造标志性记忆点

ProofArena 必须有别人一听就记住的产品记忆点。

建议形成几个固定概念：

- **解法擂台**：同一道题，不同解法正面交锋
- **思路图谱**：一题不是一个答案，而是一张推理地图
- **为什么不用**：解释被放弃的方法，训练路线判断
- **推理回放**：看见一条解法如何被发现
- **解法 Git**：解法可以 fork、diff、challenge、merge
- **数学身份**：用户主页展示真实推理作品
- **Proof Graph**：平台的底层推理网络

这些概念要反复出现在产品、文档、分享图、比赛、公开主页中，形成品牌语言。

### 13.10 高风险但高回报的功能方向

这些功能不是第一天就做完，但应作为长期想象力边界。

#### 解法 Diff

用户选择两条解法，系统不只是并排展示，而是指出：

- 共同观察是什么
- 第一个分叉点在哪里
- 哪条路线减少计算
- 哪条路线增加抽象
- 哪条路线更适合考场

#### 解法 Fork

用户可以基于一条解法创建改写版：

- 保留原解法来源
- 标注修改点
- 提交为挑战或改进
- 审核后进入正式图谱

#### 推理时间线

展示一道题从最早解法到后续挑战、改写、赛后整理的演化历史。

#### 个人风格雷达

不只是统计数量，而是分析：

- 你偏好什么入口
- 你常用什么转化
- 你的计算风险高不高
- 你的表达是否教学友好
- 你最适合挑战哪类解法

#### 题目生命力指数

一题是否值得长期展示，不只看难度，而看：

- 解法分叉数
- 挑战关系数
- 方法边界数
- 迁移题数
- 用户贡献数

#### 推理地图分享卡

分享出去的不是答案截图，而是一张“这道题的思路地图”。

这会比普通题解分享更有传播辨识度。

### 13.11 颠覆性验收标准

一个版本是否足够有开创性，不看功能数量，而看是否让用户产生以下反应：

- “我第一次看到数学题可以这样组织。”
- “这里不是告诉我答案，而是在训练我怎么选择路线。”
- “这道题我居然愿意看三条解法的对比。”
- “我想提交自己的解法去挑战它。”
- “这个个人主页比刷题记录更能代表我的数学能力。”
- “普通搜题软件给不了这种东西。”

如果用户只是觉得“这个题库做得不错”，说明还不够。

目标不是更好用的题库，而是让用户意识到旧题库范式已经过时。

## 14. 最终状态

如果改造成功，ProofArena 应该呈现出一种别人难以复制的形态：

- 它不是题库，但拥有题库的检索能力
- 它不是论坛，但拥有高质量讨论沉淀
- 它不是 AI 搜题，但拥有智能审核和结构化辅助
- 它不是竞赛平台，但拥有解法竞技和个人荣誉
- 它不是知识图谱 demo，但每条解法都在图谱中产生关系

用户最终应该产生这种感受：

> 我不是来 ProofArena 搜答案的。我是在这里训练、比较、展示和升级我的数学思维。

这就是 ProofArena 要领导的革命：让数学学习从“找答案”进入“理解推理、比较路线、创造解法”的时代。
