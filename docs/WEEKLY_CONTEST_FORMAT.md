# ProofArena 一周赛制规划

本文档描述一套适合 10-15 人小规模参赛的 ProofArena 一周赛制。目标不是把比赛做成单纯刷题计时器，而是同时考察稳定解题、快速判断、深度思考、挑战洞察和赛后讨论质量。

## 1. 赛制目标

这套赛制希望形成四种互补的竞技体验：

- **普通题**：每天稳定开放，考察持续解题能力和基础表达。
- **挑战题**：前两天开放，鼓励高难度探索，并给后续普通题提供小倍率。
- **计时题**：手动解锁后独立计时，考察快速判断和熟练度。
- **大题**：中期公布，给参赛者 2-3 天自由作答，决定深度上限。

整体节奏应当紧凑但不压迫。参赛者每天都有事可做，但不会因为错过某一个时间点就彻底失去参与价值。

## 2. 推荐时间线

周期按 8 天设计。前 5 天为主要作答期，第 6-7 天为讨论和评分期，第 8 天公布最终结果。

| 时间 | 内容 | 说明 |
| --- | --- | --- |
| Day 1 | 普通题、计时题、挑战题开放 | 挑战题开始收集倍率依据 |
| Day 2 | 普通题、计时题、挑战题继续开放 | Day 2 结束后锁定挑战题提交 |
| Day 3 | 普通题、计时题、大题开放 | 公布大题，进入深度作答阶段 |
| Day 4 | 普通题、计时题、大题作答 | 维持日常节奏 |
| Day 5 | 普通题、计时题、大题作答或截止 | 根据题目难度决定大题是否延至 Day 6 |
| Day 6 | 大题截止、公开讨论、初步评分 | 不再接收全新核心解法 |
| Day 7 | 讨论、申诉、评分修正 | 处理误判和表达补充 |
| Day 8 | 公布最终榜单和优秀思路 | 发布题解、榜单、奖项和复盘 |

普通题建议每天固定窗口：

```text
10:00 开放
17:00 截止
20:00 公布题解、当日结果和榜单
```

计时题可以在当天开放窗口内由选手自行解锁。挑战题和大题按跨日窗口处理。

## 3. 总分结构

推荐总分公式：

```text
总分 = 普通题原始分 × 挑战倍率 + 计时题分 + 大题分 + 可选奖励分 - 违规扣分
```

关键约束：

- 挑战倍率只作用于普通题。
- 计时题不吃挑战倍率。
- 大题不吃挑战倍率。
- 奖励分建议默认不进入总分，只作为荣誉；如果进入总分，上限应很低。

榜单应同时展示：

- 普通题原始分
- 挑战倍率
- 普通题结算分
- 计时题分
- 大题分
- 总分

这样可以避免参赛者只看到一个总分，却不知道差距来自速度、稳定性、倍率还是大题。

## 4. 普通题

### 4.1 定位

普通题是主赛制的日常骨架。它应该让大多数参赛者每天都能完成一部分，同时给强选手留出区分空间。

推荐每天 3 道：

| 类型 | 难度 | 目的 |
| --- | --- | --- |
| 热身题 | 中低 | 保证参与感 |
| 主线题 | 中等 | 形成主要区分 |
| 拔高题 | 中高 | 给强选手空间 |

每题建议 100 分，每天满分 300 分，5 天普通题原始满分 1500 分。

### 4.2 评分规则

| 提交情况 | 分数 |
| --- | ---: |
| 完整正确，表达清楚 | 100 |
| 方法正确但有小漏洞 | 80-95 |
| 核心思路正确，关键细节缺失 | 50-75 |
| 有有效观察，但未形成完整解法 | 20-45 |
| 只有猜测或不可验证推理 | 0-15 |
| 空交或无关内容 | 0 |

普通题应尽量按统一 rubric 评分。评审需要给出简短理由，尤其是 50 分以上但未满分的提交，应说明缺失的关键环节。

### 4.3 截止与迟交

推荐保留很小的宽限，但不要让迟交成为常态：

| 延迟时间 | 处理 |
| --- | --- |
| 0-10 分钟 | 该题得分乘 0.9 |
| 10-30 分钟 | 该题得分乘 0.7 |
| 超过 30 分钟 | 不计分 |

如果平台可以稳定控制提交窗口，也可以选择更硬的规则：截止后直接关闭提交。

## 5. 挑战题与小倍率

### 5.1 定位

挑战题在 Day 1-2 开放，难度高、数量可以较多，参赛者自行选择投入多少。它不直接贡献大量分数，而是通过小倍率影响普通题，制造前期策略选择。

挑战题允许提交：

- 关键观察
- 解题思路
- 局部引理
- 反例构造
- 接近完整的证明或算法
- 对不同路线的比较

挑战题不要求完整解出，但必须能解释为什么该思路可能有效。只有“用归纳”“看不变量”“考虑构造”这类空泛描述不应获得高分。

### 5.2 倍率档位

推荐倍率上限为 1.20x，最高不超过 1.25x。

| 挑战题表现 | 普通题倍率 |
| --- | ---: |
| 未提交或无有效内容 | 1.00x |
| 有 1-2 个合理观察 | 1.05x |
| 有明确可推进方向 | 1.10x |
| 解决了关键难点，或多个高质量 partial | 1.15x |
| 接近完整解法，或特别漂亮的核心想法 | 1.20x |

如果需要更量化，可以先给挑战题一个 0-100 分的内部评分，再映射倍率：

| 挑战评分 | 普通题倍率 |
| ---: | ---: |
| 0-19 | 1.00x |
| 20-39 | 1.05x |
| 40-59 | 1.10x |
| 60-79 | 1.15x |
| 80-100 | 1.20x |

### 5.3 挑战题评分维度

| 维度 | 分值 | 看什么 |
| --- | ---: | --- |
| 关键观察 | 30 | 是否抓住题目结构、约束或突破口 |
| 方法可行性 | 30 | 这条路线是否能真实推进，而不是只停在口号 |
| 推理清晰度 | 20 | 读者是否能理解每一步为什么成立 |
| 创造性与迁移价值 | 20 | 是否提供新角度、泛化想法或可复用模式 |

挑战题最终只产出一个倍率，不逐题叠乘。这样能防止早期优势过大，也便于解释榜单。

## 6. 计时题

### 6.1 定位

计时题是独立的速答模块。它考察快速判断、熟练度和小规模计算，不考察长证明。

题目默认锁定。参赛者手动解锁后开始计时，提交成功或提交错误时结束该题计时。

计时题只允许：

- 选择题
- 填空题

不允许证明题、长阅读题或需要大量书写的题。

### 6.2 推荐配置

| 项目 | 建议 |
| --- | --- |
| 开放时间 | Day 1-5 每天开放 |
| 每日数量 | 3 道 |
| 单题满分 | 30 分 |
| 单题限时 | 120 秒 |
| 提交次数 | 每题 1 次 |
| 答错 | 0 分 |
| 超时 | 0 分 |
| 是否吃倍率 | 否 |

5 天共 15 道计时题，总分 450 分。这个占比足够有存在感，但不会压过普通题和大题。

### 6.3 计时题计分

推荐使用阶梯计分，比连续公式更容易理解，也更能容忍小规模网络波动。

以单题 30 分、限时 120 秒为例：

| 用时 | 得分 |
| ---: | ---: |
| 0-15 秒 | 30 |
| 16-30 秒 | 26 |
| 31-60 秒 | 21 |
| 61-90 秒 | 15 |
| 91-120 秒 | 9 |
| 超过 120 秒 | 0 |

答错直接 0 分，不再按速度给分。

如果希望更连续，可以使用：

```text
得分 = 单题满分 × max(0, 1 - 用时 / 限时)
```

但正式运营中优先推荐阶梯计分。

### 6.4 计时题公平规则

计时题最容易产生争议，规则必须提前写清楚：

- 解锁后不可暂停。
- 页面刷新、断网、误关不重置计时。
- 每题只有一次提交机会。
- 答错、超时或格式不符合要求均为 0 分。
- 计时以服务器记录为准。
- 如果出现平台故障，由主办方统一决定补偿、重赛或作废。
- 统一公布答案前，不展示他人的计时题答案。

填空题需要给出格式要求：

| 答案类型 | 格式 |
| --- | --- |
| 整数 | 只填数字 |
| 分数 | 使用 `a/b`，默认要求约分 |
| 多个整数 | 使用升序逗号分隔，如 `1,3,5` |
| 字母答案 | 默认大小写不敏感，除非题目说明 |
| 表达式 | 题目必须明确等价判定方式 |

计时题应尽量避免答案等价形式过多的填空题，否则评分会变得麻烦。

## 7. 大题

### 7.1 定位

大题是整场比赛的深度核心。它不吃挑战倍率，给参赛者 2-3 天自由作答，适合考察完整证明、复杂构造、综合分析或多路线比较。

推荐 Day 3 公布，Day 5 或 Day 6 截止。

### 7.2 分值

推荐大题满分 1000 分。这样它能明显影响最终排名，又不会完全覆盖普通题和计时题。

参考总分结构：

| 模块 | 满分或上限 |
| --- | ---: |
| 普通题原始分 | 1500 |
| 普通题结算分 | 最高约 1800，按 1.20x 计算 |
| 计时题 | 450 |
| 大题 | 1000 |

这个比例下，稳定性、速度和深度都有存在感。

### 7.3 大题评分维度

| 维度 | 分值 |
| --- | ---: |
| 核心思路 | 250 |
| 关键步骤与引理 | 250 |
| 完整性与严谨性 | 250 |
| 表达清晰度 | 150 |
| 创造性与简洁性 | 100 |

参考分段：

| 提交水平 | 分数 |
| --- | ---: |
| 完整正确，表达清楚 | 900-1000 |
| 基本正确，有少量可修补漏洞 | 750-899 |
| 主线正确，但关键步骤缺失 | 550-749 |
| 有关键观察，但未完成证明 | 300-549 |
| 有局部有效想法 | 100-299 |
| 无有效进展 | 0-99 |

大题必须允许 partial credit。否则高难题很容易把大量参赛者压成 0 分，降低参与体验。

## 8. 讨论期与申诉

### 8.1 讨论期目标

Day 6-7 的讨论期不是延长答题时间，而是为了：

- 公开优秀思路
- 解释评分依据
- 发现误判和题目漏洞
- 让参赛者学习不同路线
- 产出赛后可沉淀的解法材料

### 8.2 允许内容

讨论期允许：

- 解释自己原提交中没写清楚的地方。
- 指出评审对原提交的误读。
- 补充已有思路的细节。
- 对他人解法提出问题或指出漏洞。
- 比较不同路线的优劣。

### 8.3 不允许内容

讨论期不允许：

- 提交全新核心思路并要求重新按完整解法评分。
- 在看到别人方案后补成完整解法。
- 私下交换答案后再公开包装成个人补充。
- 对他人进行人身攻击或干扰讨论。

### 8.4 加分与修正

推荐规则：

```text
讨论补充最多增加原大题分的 5%。
申诉修正不设上限，但必须证明原提交本来就应得这些分。
```

也就是说，补充说明有上限，纠正误判无上限。

## 9. 奖项与荣誉

10-15 人规模下，奖项要少而清楚。建议默认只作为荣誉，不进入总分。

可设置：

- 全场总冠军
- 普通题稳定王
- 计时题速答王
- 大题最佳解法
- 最佳挑战思路
- 最佳讲解
- 最强 partial
- 最佳讨论贡献

如果奖励分进入总分，建议总上限不超过 50-100 分，并在赛前明确说明。

## 10. 公平性与工具规则

### 10.1 协作规则

推荐默认规则：

- 截止前不得私下交换具体答案、完整证明或关键路线。
- 可以讨论题意、平台问题和格式问题。
- 引用他人想法必须署名。
- 讨论期可公开交流，但新增核心思路不计入原始提交评分。

### 10.2 AI 与资料规则

必须在赛前明确 AI 和外部资料边界。可选两种模式。

严格模式：

```text
普通题和计时题禁止 AI 辅助。
大题可查阅公开资料，但必须注明来源。
所有提交必须能由本人解释。
```

开放模式：

```text
允许使用工具查资料、验算或整理表达。
最终思路必须由本人理解并能解释。
无法解释的提交按无效处理。
```

考虑到计时题强调速度，建议无论采用哪种模式，计时题都禁止外部辅助。

### 10.3 违规处理

| 行为 | 处理建议 |
| --- | --- |
| 截止前私下交换答案 | 相关题目 0 分，严重者取消资格 |
| 抄袭他人思路不署名 | 扣分或取消相关提交 |
| 计时题试图多次作答或绕过计时 | 该题 0 分，严重者取消资格 |
| 讨论期抄袭他人补交 | 新增内容不计分 |
| 恶意攻击或干扰讨论 | 警告、扣分或禁言 |

## 11. 榜单设计

榜单实时公开，但应拆分展示，避免一个总分造成黑箱感。

推荐榜单字段：

| 字段 | 说明 |
| --- | --- |
| 排名 | 当前总分排序 |
| 选手 | 昵称 |
| 普通题原始分 | 未乘倍率前的普通题总分 |
| 挑战倍率 | 1.00x-1.20x |
| 普通题结算分 | 原始分乘倍率 |
| 计时题分 | 速答模块得分 |
| 大题分 | 大题评分完成后展示 |
| 总分 | 当前结算总分 |

比赛期间可以实时更新普通题、计时题和倍率。大题在评审完成前可以显示为 `评分中`，避免未校准分数影响讨论。

## 12. 题目设计建议

### 12.1 普通题

普通题要有稳定可评的标准答案，避免过度开放。每天 3 题中至少 1 题应让大多数参赛者有把握拿到分数。

适合：

- 中等难度证明题
- 有清晰答案的构造题
- 小规模分类讨论
- 可以 partial credit 的推理题

不适合：

- 依赖大量查资料的题
- 结论含糊的开放题
- 评分标准很难统一的长论文式题目

### 12.2 挑战题

挑战题可以更难、更开放，甚至不要求官方完整解法一开始就非常短。但评审必须知道如何判断 partial 的价值。

适合：

- 难证明题的关键引理
- 多路线探索题
- 反例或构造方向
- 高难不变量或极值问题

### 12.3 计时题

计时题必须短、准、可自动判分。

适合：

- 判断命题真伪
- 小规模计算
- 选择正确不变量
- 找最小反例
- 识别哪一步推理错误

不适合：

- 长题干
- 多等价答案填空
- 需要复杂化简的表达式
- 容易因为格式而误判的题

### 12.4 大题

大题要能承载讨论价值。即使参赛者做不完，也应能产生有意义的 partial。

适合：

- 综合证明题
- 多解法比较题
- 深构造题
- 概念边界题

## 13. 平台功能需求

为了完整支持该赛制，平台需要逐步具备以下能力。

### 13.1 数据模型

比赛题目需要区分：

- 普通题
- 挑战题
- 计时题
- 大题
- 讨论题

建议为 contest problem 增加或抽象出字段：

```text
problem_phase: daily | challenge | sprint | major | discussion
score_max
score_policy
multiplier_eligible
timed_mode_enabled
time_limit_seconds
max_attempts
```

挑战倍率可以存为选手级 contest profile：

```text
contest_slug
user_id
challenge_score
challenge_multiplier
multiplier_reason
```

计时题提交需要记录：

```text
unlock_at
submitted_at
elapsed_ms
is_correct
raw_answer
normalized_answer
score
```

### 13.2 后台能力

管理员需要能够：

- 配置题目类型和开放窗口。
- 设置计时题限时、选项、标准答案和格式规则。
- 查看每个选手的计时题解锁与提交时间。
- 给挑战题评分并生成倍率。
- 给普通题和大题人工评分。
- 处理申诉和评分修正。
- 锁定最终榜单。

### 13.3 前台能力

参赛者需要能够：

- 查看今日普通题。
- 查看挑战题开放状态和自己的倍率状态。
- 手动解锁计时题，并看到剩余时间。
- 提交选择题或填空题答案。
- 查看自己的原始分、倍率、计时题分和总分。
- 在讨论期查看公开解法、评分理由和讨论内容。

## 14. 运营检查清单

赛前：

- 确认赛程、题目数量、开放窗口和截止时间。
- 确认挑战倍率 rubric 已公布。
- 确认计时题答案格式已写清。
- 确认大题 partial credit rubric 已公布。
- 确认 AI、外部资料和协作规则。
- 试跑一题计时题，检查解锁、计时、提交、判分链路。
- 准备榜单展示模板。

比赛期间：

- 每天检查提交是否正常。
- 每晚公布普通题题解和当日结果。
- 及时处理平台问题，不临时改变核心评分规则。
- 记录有价值的思路，供讨论期整理。

讨论期：

- 公布大题初评分和评分理由。
- 收集申诉。
- 公开优秀思路。
- 修正误判。
- 锁定最终分数。

赛后：

- 公布最终榜单。
- 发布单项奖。
- 整理优秀解法进题库。

## 15. 推荐最终版本

第一版建议使用以下参数：

```text
周期：8 天
参赛规模：10-15 人

普通题：
Day 1-5 每天 3 道
每题 100 分
当天上午开放，下午截止，晚上公布

挑战题：
Day 1-2 开放
只评一个普通题倍率
倍率范围 1.00x-1.20x
只作用于普通题

计时题：
Day 1-5 每天 3 道
每题 30 分
手动解锁后开始计时
限时 120 秒
选择题或填空题
答错或超时 0 分
不吃倍率

大题：
Day 3 公布
Day 5 或 Day 6 截止
满分 1000 分
不吃倍率

讨论评分：
Day 6-7 公开讨论、申诉、修正评分
讨论补充最多增加原大题分的 5%
申诉修正不设上限，但必须基于原提交

结果：
Day 8 公布最终榜单、单项奖和优秀思路
```

这套配置的核心优点是：普通题保证每日参与，挑战题提供前期策略，计时题增加竞技兴奋点，大题保留深度上限，讨论期负责学习和公平。
