# 内容与数据规范

ProofArena 的核心资产不是题目数量，而是可学习、可比较、可复核的解法内容。

## 一道题的最低标准

每道正式题目至少需要：

- 可核对的来源
- 完整题干
- 明确答案
- 难度、题型与专题标签
- “这题怎么想到”学习指南
- 至少两条有实质差异的解法
- 每条解法的评分理由
- 验证状态和检查项

两条解法若只有变量名、书写顺序或少量代数变形不同，不应作为独立路线。

## Problem 字段

| 字段 | 说明 |
| --- | --- |
| `id` | 稳定路由 ID，例如 `tj-2026-18` |
| `year` | 考试年份 |
| `region` | 卷别，受 `ExamRegion` 类型约束 |
| `paper` | 科目或试卷名称 |
| `number` | 面向用户的题号 |
| `difficulty` | 基础 / 中档 / 压轴 |
| `questionType` | 单选 / 多选 / 填空 / 解答 |
| `tags` | 可用于筛选的专题标签 |
| `title` | 能概括数学任务的编辑标题 |
| `statement` | 按自然段拆分的题干 |
| `answer` | 简明最终结论 |
| `heat` | Demo 展示热度，不代表官方难度 |
| `sourcePdf` | 原题来源文件路径 |
| `sourcePage` | PDF 页码，从 1 开始 |
| `answerPdf` | 可选答案文件 |
| `learningGuide` | 解题前的观察与推荐路径 |
| `solutions` | 解法数组 |

## LearningGuide 字段

### `observation`

告诉学生读题时先看什么。应落到具体条件，而不是写“认真审题”。

好例子：

> 目标不是交点坐标本身，而是从上顶点连到两个交点的斜率比。

### `triggers`

使用“条件 → 联想”的短句，建立可迁移模式。

好例子：

> 直线与圆相切 → 圆心到直线距离 = 半径

### `recommendation`

解释不同学习目标应先看哪条解法。

## Solution 字段

| 字段 | 说明 |
| --- | --- |
| `id` | 题目内唯一、可作为锚点 |
| `title` | 清楚说明路线，而非“解法一” |
| `author` | 作者或整理来源 |
| `authorRole` | 考场标准解、结构路线等定位 |
| `tags` | 该解法自身的技术标签 |
| `badge` | 卡片上显示的简短优势 |
| `origin` | 为什么会想到这条路线 |
| `keyTransform` | 决定后续推导的核心转化 |
| `summary` | 完整、顺序明确的步骤数组 |
| `scores` | 五维评分 |
| `scoringReason` | 解释优势、代价和分差 |
| `verification` | 验证状态、工具和检查项 |
| `estimatedMinutes` | 熟练学生完成所需的大致时间 |

## 完整过程写法

`summary` 不是提纲。每一步都应让读者知道：

1. 使用了什么条件或定理
2. 得到了什么中间结论
3. 为什么可以进入下一步

不要写：

> 计算可得答案。

应写：

> 将切线代入椭圆得 $5x^2-16x+12=0$，两根对应交点横坐标 $2$ 与 $6/5$。

## 验证状态

- `verified`：有明确、可重复的机器或形式化验证
- `partial`：部分关键步骤经过机器检查，其余仍依赖人工证明
- `manual`：由人工对照题面、官方答案或独立复算

不要为了页面效果把人工复核标成 CAS verified。

## LaTeX 规则

- 行内公式写成 `$...$`
- 在 TypeScript 字符串中将 `\` 写成 `\\`
- 使用 `\\dfrac` 提升分式可读性
- 中文标点放在数学环境之外
- 长公式应拆成多个步骤，避免移动端难以阅读

## 来源与版权

- 题目来源必须可核对
- 不大段复制第三方商业解析
- 原始试卷与答案的权利归原权利人
- 原创整理、学习指南、评分理由和解法表达按 CC BY-SA 4.0 提供
- 投稿者必须确认自己有权授权提交内容

## 内容 PR 建议

一个内容 PR 最好只处理：

- 一道新题
- 一条新解法
- 一组明确的纠错
- 一次评分规则校准

PR 中附上人工复核过程和至少一个可验证检查。
